## Friction observed (ranked, most-painful first)

For each item: what happened, why it slowed the walk, whether fixed in-budget or deferred.

1. **A-side fields rendered as '-'.** The merge-candidates review screen showed the candidate column entirely blank because `_load_person` only queried the `persons` table, while `candidate_a_id` typically references `candidate_persons` (Phase 5.1 dual-table reality). Phase 5 UI never picked up the Phase 5.1 read-side. *Status:* fixed in-budget (see Fix #1).
2. **Evidence column shows "(no citation linked to candidate)" for every candidate.** Same Phase 5.1 read-side gap: `render_left` queried `entity_citations` which only holds canonical-side entries. Candidate rows carry `chunk_id` + `quote` directly. *Status:* fixed in-budget (see Fix #2).
3. **No way to see what merges have been committed.** Curator wanted a history of decisions during the walk to verify what landed and spot accidents. *Status:* fixed in-budget (see Fix #3) — sidebar "Recent decisions" panel reads `audit_log` and shows the last 20 merge/reject/split/edit rows.
4. **Edit & accept's second click invisible.** Same button label for "enter edit mode" and "commit edits" — curator didn't realize they had to click `e` again. Also surfaced a deferred Phase 5 schema migration: live DB's `audit_log` CHECK constraint hadn't been migrated to include `'edit'`, so commit failed with IntegrityError. *Status:* DB migration ran as break-fix (commit `b4d62c1`); button label flips to "✓ Confirm edits" in edit mode (Fix #4, ~10 LOC — over the original ≤3 budget by 1 small polish; the budget intent was "no new features," and this is clarity not features).
